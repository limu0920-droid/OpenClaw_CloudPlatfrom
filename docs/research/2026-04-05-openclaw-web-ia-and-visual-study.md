# OpenClaw Web 端调研结论

## 1. 研究问题

- 这次研究支持什么决策：确定 OpenClaw 平台 Web 端第一阶段应该先做什么、页面如何拆、视觉风格如何定、Mock API 应该围绕哪些实体建模。
- 范围：
  - 本地 `docs/` 中的需求文档、DDL 初稿、部署样例
  - OpenClaw 官方公开页面与 Control UI 能力说明
  - `yunwu.ai/console`、`co-cdn.yes.vg`、`openclaw.feishu.cn` 三个参考站点公开可见页面

## 2. 结论先行

- 第一阶段 Web 端核心不是商城，而是“实例控制台”。
- 产品形态应拆成两套体验：
  - `portal`：租户用户视角，负责实例、访问入口、配置、备份、任务与告警摘要。
  - `admin`：平台运维视角，负责租户、实例、运行态、任务、告警与审计。
- 研发形态建议先做一个 Vue 应用，使用 `/portal` 与 `/admin` 双布局并共享组件、Mock 类型和状态枚举，便于快速迭代。
- 后端先用 Go 提供 Mock API，前端不直接内嵌大段静态 JSON，保证后续切真接口时改动最小。
- 视觉策略不应照搬某个参考站点，更合理的组合是：
  - `飞书 OpenClaw` 的企业可信感
  - `YesCode` 的工作流与步骤表达
  - `云雾` 的登录与接入入口轻量感

## 3. 关键事实

### 3.1 本地需求文档事实

- 文档明确要求平台前端不要只复用 OpenClaw 原生 Control UI，而要拆为“用户侧门户 + 平台管理后台”。
- 第一阶段最重要的业务主线是：
  - 实例生命周期
  - 配置读取与发布
  - 备份与恢复
  - 异步任务跟踪
  - 告警与审计
- 数据模型初稿已经覆盖第一阶段 Web 端所需的大多数实体：
  - `tenant`
  - `user_account`
  - `service_instance`
  - `instance_access`
  - `instance_config`
  - `backup_record`
  - `operation_job`
  - `operation_log`
  - `alert_record`
  - `cluster / cluster_node / runtime_container`
- 部署样例已经给出建议拓扑：
  - `portal-web`
  - `admin-web`
  - `platform-api`
  - `job-worker`
  - `runtime-agent`
  - `openclaw-gateway`

### 3.2 OpenClaw 原生能力事实

- OpenClaw 原生控制面更偏“运行调试与运维控制”，覆盖聊天调试、渠道、会话、定时任务、技能、节点、审批、配置、健康、日志与更新等能力。
- 这说明平台第一阶段不应该试图完整重写原生 Control UI，而应先完成平台门户层与控制平面层的核心闭环。

### 3.3 参考站点事实

#### `yunwu.ai/console`

- 公开可见的是登录页，不是登录后的控制台。
- 页面采用左右分栏：左侧品牌展示，右侧登录表单。
- 视觉特征是亮色、蓝白主色、大圆角、轻量企业 SaaS 感。

#### `co-cdn.yes.vg`

- 公开可见的是产品首页，不是工作台。
- 页面是深色开发者产品风格，核心表达是 Hero + 工作流演示卡 + 指标卡 + 三步上手。
- 视觉特征是深灰黑底、亮蓝强调、半透明卡片、流程导向。

#### `openclaw.feishu.cn`

- 公开可见的是飞书 OpenClaw 官网落地页。
- 页面强调“一键部署、原生体验、企业级安全”三条价值主张。
- 视觉特征是亮底、规整中文排版、飞书蓝按钮、企业可信感较强。

## 4. 主要分析

### 4.1 第一阶段功能边界

#### `portal` 第一阶段

- 登录页
- 工作台首页
- 实例列表
- 实例详情
  - 概览
  - 访问入口
  - 配置中心
  - 备份记录
- 任务中心
- 操作记录

#### `admin` 第一阶段

- 平台总览
- 租户列表与详情
- 实例管理
- 实例详情
  - 概览
  - 运行信息
  - 配置摘要
  - 备份记录
  - 操作日志
- 任务中心
- 告警中心
- 审计日志

### 4.2 不建议首批纳入的范围

- 支付、订单、退款、发票
- 复杂审批流
- 远程终端
- 容器级实时诊断
- 产品与套餐管理
- 深度监控图表与容量治理

### 4.3 信息架构建议

#### `portal`

- 一级导航：
  - 工作台
  - 实例
  - 任务
  - 日志
- 首页模块：
  - 租户欢迎卡
  - 实例概览
  - 最近任务
  - 告警摘要
  - 访问入口快捷区
  - 配置与备份快捷区

#### `admin`

- 一级导航：
  - 总览
  - 租户
  - 实例
  - 任务
  - 告警
  - 审计
- 首页模块：
  - 全局健康概览
  - 实例与租户分布
  - 任务队列
  - 重点告警
  - 异常实例

### 4.4 视觉方向建议

- 双壳策略：
  - `portal` 用亮色基底，强调可信、清晰、易上手
  - `admin` 用深色控制台外壳，强调状态流、运维效率、信息密度
- 统一品牌令牌：
  - 单一企业蓝作为强强调色
  - 大圆角卡片
  - 简洁中文文案
  - 卡片化信息分组
  - 清晰的主操作按钮
- 推荐风格关键词：
  - `portal`：企业、安心、开箱即用、低学习成本
  - `admin`：控制面、编排、诊断、状态流、审计

### 4.5 技术路线建议

- 前端：
  - Vue 3
  - TypeScript
  - Vue Router
  - Pinia
  - 自定义设计令牌与基础组件
- 后端：
  - Go
  - 标准库 + 轻量 HTTP 路由
  - 统一 `/api/v1` Mock 接口
- 目录策略：
  - `apps/web`
  - `apps/mock-api`
  - `docs/research`
  - `docs/plans`

## 5. 风险与待验证项

- 当前三个参考站点公开可见部分以登录页或营销页为主，未覆盖真实后台 IA，因此后台结构判断主要来自产品抽象而不是页面复刻。
- 还需要在后续阶段确认：
  - `portal` 与 `admin` 最终是否拆为两个独立应用
  - 是否需要接入 OpenClaw 原生 Control UI 的嵌入或跳转
  - 多租户模式最终选择“共享实例逻辑隔离”还是“租户独立实例”

## 6. 下一步

- 先创建实施计划文档，明确文件结构和开发顺序。
- 先完成一套可运行的 Vue Web 骨架，覆盖 `/portal` 与 `/admin` 主页面。
- 同步提供 Go Mock API，优先返回概览、实例、配置、备份、任务、告警相关数据。
- 等前后端打通后，再决定第二阶段是否拆应用、引入真鉴权、补图表与审批流。
